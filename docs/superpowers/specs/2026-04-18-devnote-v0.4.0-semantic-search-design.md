# DevNote v0.4.0 — Semantic Search Design Spec

**Date:** 2026-04-18
**Status:** Approved
**Scope:** v0.4.0 only — second release of Phase 2 (AI memory arc)

---

## One-line summary

Add semantic search over all past notes via Gemini `text-embedding-005` embeddings, backed by a lazy-spawned Python subprocess that handles vector math. TypeScript orchestrates, Python executes the AI/ML layer.

---

## The shift

**v0.3.0:** DevNote remembers every note locally (SQLite), but searching requires knowing the exact title.
**v0.4.0:** DevNote searches by **meaning**, not keywords. Type "notion rate limit thing" → find it even if those exact words aren't in the title.

This is the release where DevNote stops being *browsable memory* and starts being *queryable memory*.

---

## Decisions (D1–D14)

| # | Decision | Choice |
|---|---|---|
| D1 | TS ↔ Python integration | Child process + stdin/stdout JSON-RPC |
| D2 | Python deployment | User installs Python 3.10+, DevNote manages a venv in `globalStorageUri` |
| D3 | Python packages | Minimal: `google-generativeai==0.8.3` + `numpy==2.1.3`, exact pinning |
| D4 | Similarity algorithm | Dot product (equivalent to cosine on L2-normalized vectors, 3x faster) |
| D5 | Backfill strategy | Batch backfill at first search, with popup using Gemini batch API |
| D6 | Embedding failure handling | Synchronous embed during sync, NULL fallback on failure (best-effort) |
| D7 | Search input location | Above Recent Notes list; list switches between Recent and Results based on query state |
| D8 | Row layout | Title + branch + relative date + percentage similarity score |
| D9 | Search mode cues | Morphing header, ✕ clear affordance, empty & loading states |
| D10 | Score format | Percentage inline with date (e.g. `87% · 2h`) |
| D11 | k + threshold | k=5, threshold=0.35 (drops noise below, preserves weak-but-plausible) |
| D12 | Schema migration | None — v0.3.0 already pre-baked `embedding` + `embedding_model` columns |
| D13 | Model-change detection | Write `embedding_model` column in v0.4.0, defer active drift UI to v0.4.x |
| D14 | Subprocess spawn timing | Lazy — only spawn Python worker on first search click |

### Layered responsibility

| Layer | Owns | Doesn't know about |
|---|---|---|
| `worker.py` | Embedding API calls, numpy vector math, in-memory matrix | SQLite, UI, git |
| `PythonBridge.ts` | Subprocess lifecycle, JSON-RPC framing, crash recovery | What messages mean semantically |
| `SearchService.ts` | Typed search operations: `embedNote`, `searchQuery`, `batchEmbedAll`, `ensureReady` | Subprocess transport details |
| `SidebarProvider.ts` | UI state machine, webview messaging | Python directly |

### PythonBridge lifecycle

- **Spawn:** lazy, on first search click (not on extension activation)
- **Venv setup:** runs BEFORE spawn if `venvReady` flag missing in globalState
- **Crash recovery:** auto-respawn up to 3 times per session, then escalate to the user
- **Shutdown:** `subprocess.kill()` on extension `deactivate()` — no orphan Python processes
- **Protocol:** newline-delimited JSON. Request: `{id, method, params}`. Response: `{id, result}` or `{id, error}`.

### SearchService API

```typescript
class SearchService {
  async ensureReady(): Promise<void>;                       // venv check + spawn + backfill
  async embedNote(text: string): Promise<void>;             // during sync flow
  async searchQuery(q: string, k: number): Promise<SearchResult[]>;
  async batchEmbedAll(notes: Array<{id: string, text: string}>): Promise<void>;
}
```

## Data Model

### Schema (unchanged from v0.3.0)

```sql
CREATE TABLE notes (
  id               TEXT PRIMARY KEY,
  title            TEXT NOT NULL,
  branch_name      TEXT NOT NULL,
  content_json     TEXT NOT NULL,
  content_markdown TEXT NOT NULL,       -- embedding input source
  notion_page_id   TEXT,
  notion_page_url  TEXT,
  created_at       INTEGER NOT NULL,
  embedding        BLOB,                -- FILLED in v0.4.0 (was NULL in v0.3.0)
  embedding_model  TEXT                 -- FILLED with "text-embedding-005"
);

CREATE INDEX idx_notes_created_at ON notes(created_at DESC);

CREATE TABLE schema_version (
  version    INTEGER PRIMARY KEY,
  applied_at INTEGER NOT NULL
);
```

**`schema_version` stays at 1. No migration #2. No `ALTER TABLE`.**

### New queries MemoryStore adds (4 total)

```sql
-- 1. Single embed after sync
UPDATE notes SET embedding = ?, embedding_model = ? WHERE id = ?;

-- 2. Notes needing backfill (first search)
SELECT id, content_markdown FROM notes WHERE embedding IS NULL;

-- 3. Warm-load matrix on worker startup
SELECT id, embedding FROM notes WHERE embedding IS NOT NULL;

-- 4. Backfill-needed fast path
SELECT COUNT(*) FROM notes WHERE embedding IS NULL;
```

### Embedding format

- Gemini `text-embedding-005` returns 768 float values, L2-normalized
- Stored as raw bytes: `Float32Array(768).buffer` = 3072 bytes per row
- Per-note total (content + embedding): ~6 KB
- 10,000 notes = ~60 MB on disk

### Python worker in-memory structure

```python
class VectorStore:
    ids: list[str]          # N UUIDs, ordered
    matrix: np.ndarray      # shape (N, 768), dtype=float32
    model: str              # "text-embedding-005"
```

Loaded once at worker startup via query #3, appended when new notes arrive.

### New TypeScript type

```typescript
export interface SearchResult {
  id: string;
  title: string;
  branchName: string;
  createdAt: number;
  score: number;  // 0.0–1.0, rendered as "87%" in UI
}
```

## User Flows

### Flow 1: Upgrade from v0.3.0, user never uses search

Extension activates silently. No Python spawn. No popups. Generate/sync works normally. Each sync attempts embed; if Python not ready, gracefully skips and logs a toast. DevNote behaves identically to v0.3.0 for this user forever.

### Flow 2: First-ever search click (the onboarding moment)

```text
User clicks 🔍 → types "sqlite" → 300ms debounce
        ↓
SearchService.ensureReady()
        ├── Python 3.10+ exists?
        │     NO → popup "Install Python 3.10+ [Guide]" → wait for user
        ├── Venv exists?
        │     NO → popup "Setting up Python env (~30s)" → create venv + pip install
        ├── Spawn worker.py (if not running)
        ├── Backfill needed (COUNT NULL > 0)?
        │     YES → popup "Indexing {N} past notes (~{T}s)" → batch embed
        └── Run search: embed query → dot product → top-5 > 0.35 → return
        ↓
Render "5 results for 'sqlite'" with 87% badges
```

**At most two popups, one time per user, ever.**

### Flow 3: Normal search (warm path, ~500ms)

User types → 300ms debounce → SearchService.searchQuery → worker embeds query (~200ms) → dot product against matrix (~5ms for 10K vectors) → filter >0.35 → sort → return top-5 → webview renders with morphing header.

### Flow 4: Clearing search

User clicks ✕ (or deletes query text) → `refreshIdleState()` → header morphs back to "Recent Notes" → list re-renders from MemoryStore.

### Flow 5: Empty results (all scores below threshold)

Show: *"No strong matches for 'xyz'. Try different keywords or click ✕ to go back."*

### Flow 6: Generate + sync a new note (sync now includes embed)

Existing v0.3.0 flow, plus one step after sync success:

```text
If Python worker ready → SearchService.embedNote(content_markdown) → UPDATE BLOB → toast "Note synced."
Else → skip → toast "Note synced. Search indexing on your next search."
```

### Flow 7: Click a search result

Same handler as Recent Notes row click. Opens inline Preview (v0.3.0's historical preview mode). Back button returns to the search results with the query preserved — user keeps context.

### State transitions

```text
IDLE / Recent Notes ⇄ SEARCH ACTIVE ⇄ PREVIEW (historical, query preserved)
```

Search mode is an extension of Idle, not a new top-level state. Preview state reused identically from v0.3.0.

### Sidebar layout (Idle with search)

## Error Handling

### Guiding principle

> **Notion is authoritative. Memory is best-effort. Embedding is best-effort. Search is optional.** Each new AI-layer failure lives one tier softer than the previous — never overrides essential layers.

### Error matrix (12 modes)

| Error | Severity | User sees | Extension behavior |
|---|---|---|---|
| Python not installed | Critical (search only) | Modal popup w/ install guide | Search disabled; v0.3.0 flows work |
| Python version < 3.10 | Critical (search only) | Modal popup w/ upgrade guide | Search disabled |
| Venv setup fails | User-action | Toast + retry button | Venv not created; retry works |
| User cancels setup popup | User-action | Silent | Search disabled gracefully |
| Worker spawn fails | Critical | Toast + retry | Search disabled; v0.3.0 unaffected |
| Worker crashes mid-operation | Non-critical | Toast "worker restarted" | Auto-respawn up to 3x per session |
| Embed API fails during sync | Non-critical | Toast "indexing deferred" | NULL embedding, backfill retries later |
| Embed API fails during backfill | Non-critical | Toast "indexed M/N" | Partial persists, NULLs retry next search |
| Embed API fails during query | Non-critical | Empty state "try again" | Recent Notes browsing unaffected |
| Note exceeds 2048 tokens | Edge case | Toast "too long for search" | Permanent NULL; search skips that row |
| SQLite BLOB corruption | Rare | Silent fallback | Skip bad row in matrix load; Settings → Reset |
| User offline during query | Expected | Empty state "requires internet" | Recent Notes still works (local) |

### Priority ladder (extended from v0.3.0)

```text
Notion sync succeeds?
├── NO  → Sync error state (red, draft saved)
└── YES → Memory save succeeds?
          ├── NO  → Success + toast warning
          └── YES → Embed succeeds?
                    ├── NO  → Success + toast "indexing deferred"
                    └── YES → Success state (green)
```

### PythonBridge crash recovery

Bounded respawn: up to 3 crashes per extension-activation session. After 3, search disables for the session and user sees actionable error (restart VS Code or check logs).

### New Settings buttons (power-user recovery)

- **Reset Python environment** — deletes venv, clears flags, next search re-triggers setup
- **Re-index all notes** — sets all embeddings NULL, next search re-triggers full backfill

Both gated behind confirmation popups (same pattern as Clear Memory).

## Testing Strategy

### 1. Python worker tests — `python/tests/test_worker.py` (~10 tests)

Pytest-based. Covers: embed_text (mocked API), batch_embed_texts, VectorStore.load_from, search with top-k + threshold, empty-result case, JSON-RPC happy + error paths, Q4 math sanity (dot ≡ cosine on normalized vectors).

### 2. PythonBridge tests — `test/PythonBridge.test.cjs` (~5 tests)

Uses `test/fixtures/mock_worker.py` (tiny Python echo script — no real Gemini). Covers: spawn, roundtrip message, concurrent requests, error propagation, clean shutdown.

### 3. MemoryStore delta tests — `test/MemoryStore.test.cjs` (+4 tests)

Appends to existing v0.3.0 file: UPDATE embedding BLOB round-trip, embedding_model column write, SELECT WHERE NULL returns expected rows, SELECT all embeddings matrix-load format.

### 4. Manual E2E checklist — `docs/reference/v0.4.0-test-checklist.md`

~40 checks across setup/upgrade, search flow, sync flow, error scenarios, and Settings.

### What we do NOT test in CI

- Real Gemini API calls (mocked or stubbed)
- Pyodide / WASM paths (not used)
- Cross-platform venv activation (manual verification across win/mac/linux)
- RAG / LLM synthesis flows (deferred to v0.5.0)

## What v0.4.0 is NOT

- NOT chat / RAG queries (v0.5.0)
- NOT MCP memory server (parked for post-v0.5.0 review)
- NOT chunking — notes fit in one embedding (400-1000 tokens per note, 2048 limit)
- NOT hybrid search (BM25 + semantic) — pure semantic is enough at our scale
- NOT reranking with cross-encoders — top-5 over 10K is already precise enough
- NOT an ANN index (HNSW, FAISS, sqlite-vec) — brute-force wins below 100K vectors
- NOT bundled Python — user installs their own 3.10+
- NOT auto-install via uv/micromamba — deferred; re-evaluate if user friction proves real
- NOT a query_cache table — in-memory LRU is sufficient
- NOT changes to v0.3.0 Recent Notes / Preview / Clear Memory / Draft flows
