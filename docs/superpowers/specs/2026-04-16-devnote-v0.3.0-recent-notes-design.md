# DevNote v0.3.0 — Recent Notes Design Spec

**Date:** 2026-04-16
**Status:** Approved
**Scope:** v0.3.0 only — first release of Phase 2 (AI memory arc)

---

## One-line summary

Add local SQLite storage to DevNote, capture every synced note there, and build a sidebar list to browse and re-read past notes without leaving VS Code.

---

## The shift

**v0.2.0:** DevNote is a pipe. Generate note → sync to Notion → forget locally.
**v0.3.0:** DevNote is a pipe with a notebook. Generate → sync to Notion → also write to local SQLite → never forget.

Notion remains the human-readable archive (shareable, mobile, team-facing). SQLite becomes DevNote's own memory — machine-readable, fast, offline, and the foundation for v0.4.0 search and v0.5.0 RAG/MCP.

---

## Decisions (D1–D9)

| # | Decision | Choice |
|---|---|---|
| D1 | Data source for Recent Notes | Local cache — capture on sync success to SQLite |
| D2 | Settings action | "Clear all memory" button in Settings |
| D3 | Schema foundations | MCP-friendly: full content, UUID IDs, embedding column from day 1, migrations system |
| D4 | Row fields | Title + branch name + relative date (2-line rows) |
| D5 | Click action | Inline Preview from SQLite + subtle "Open in Notion" link at bottom |
| D6 | Layout | Recent list inline in Idle state, below Generate Doc button |
| D7 | Display count | All notes, scrollable. Friendly empty state on day one. |
| D8 | Visibility | Recent list visible only in Idle state, hidden during other states |
| D9 | Clear memory flow | Settings-only location, safe confirmation popup, opt-in export-before-clear |

## Architecture

### New file (1)

- **`src/MemoryStore.ts`** — Wraps sql.js, manages devnote.db lifecycle, runs migrations, exposes a typed API. The only file that knows SQLite exists.

### Files that change (5)

- **`src/extension.ts`** — Instantiate MemoryStore, await init(), pass to SidebarProvider.
- **`src/SidebarProvider.ts`** — Accept MemoryStore in constructor. Save note to memory after sync success. New handlers for Recent list clicks and Clear All flow. ~100 new lines.
- **`webview/sidebar.html`** — Recent Notes section in Idle state. "Clear all memory" in Settings. Confirmation popup.
- **`webview/sidebar.js`** — Render Recent Notes rows, handle row clicks, render clear-memory popup. ~80 new lines.
- **`webview/sidebar.css`** — Styling for list items, popup, destructive button.

### New dependency (1)

- **`sql.js`** (~1 MB WASM) — Pure WASM SQLite. No native bindings. Same binary on Windows/macOS/Linux.

No other new deps. UUID v4 via native `crypto.randomUUID()`. Sorting by `created_at` column.

### Small change to existing service (1)

- **`src/NotionService.ts`** — `push()` currently returns `Promise<void>`. Change to return `Promise<{ pageId: string, pageUrl: string }>` by extracting these from the Notion API response (the POST response already contains both fields, we just ignore them today). Needed so SidebarProvider can pass `notionPageId` and `notionPageUrl` to `memoryStore.saveNote()`. ~5 lines changed.

### What does NOT change

- `GitService.ts` — unchanged
- `LLMService.ts` — unchanged
- `ConfigService.ts` — unchanged
- `DraftStore.ts` — unchanged
- All existing UI states — unchanged
- The generate → preview → sync flow — unchanged (one line added at the end, plus push() return value captured)

### MemoryStore API

```typescript
class MemoryStore {
  async init(): Promise<void>
  // Load sql.js WASM, open/create devnote.db, run pending migrations

  async saveNote(note: StructuredNote, meta: NoteMetadata): Promise<string>
  // Insert row, persist to disk, return UUID

  async getRecentNotes(): Promise<RecentNoteRow[]>
  // All notes as light rows (id, title, branchName, createdAt), newest first

  async getNoteById(id: string): Promise<FullNote | null>
  // Full note for inline preview

  async clearAll(): Promise<void>
  // DELETE FROM notes, persist

  async exportAll(): Promise<string>
  // JSON string of all notes for backup
}
```

### sql.js lifecycle

1. **Startup:** `initSqlJs({ locateFile })` loads the WASM file (~50-200ms, once).
2. **Open/create:** Read `devnote.db` from `globalStorageUri`. If file exists, load bytes into `new SQL.Database(bytes)`. If not, create fresh in-memory DB.
3. **Migrations:** Check `schema_version` table, run any unapplied migrations.
4. **Hold in memory:** The `SQL.Database` instance lives for the extension's lifetime.
5. **Persist on write:** After every INSERT or DELETE, call `db.export()` → `fs.writeFile()`. Reads never hit disk.

### Database file location

```typescript
const dbPath = vscode.Uri.joinPath(context.globalStorageUri, 'devnote.db').fsPath;
```

One DB across all repos. User-level, not workspace-level. Cleaned up on extension uninstall.

## Data Model

### Schema (version 1)

```sql
CREATE TABLE notes (
  id               TEXT PRIMARY KEY,     -- UUID v4
  title            TEXT NOT NULL,
  branch_name      TEXT NOT NULL,
  content_json     TEXT NOT NULL,        -- StructuredNote as JSON
  content_markdown TEXT NOT NULL,        -- serialized markdown
  notion_page_id   TEXT,                 -- nullable
  notion_page_url  TEXT,                 -- nullable
  created_at       INTEGER NOT NULL,     -- unix ms
  embedding        BLOB,                 -- NULL in v0.3.0, Float32Array(768) in v0.4.0+
  embedding_model  TEXT                  -- NULL in v0.3.0
);

CREATE INDEX idx_notes_created_at ON notes(created_at DESC);

CREATE TABLE schema_version (
  version    INTEGER PRIMARY KEY,
  applied_at INTEGER NOT NULL
);
```

### Why each column

| Column | Purpose | First used |
|---|---|---|
| `id` | Stable identity (UUID v4) | v0.3.0 |
| `title` | Display in list + future search | v0.3.0 |
| `branch_name` | Display in list row (2nd line) | v0.3.0 |
| `content_json` | Render inline Preview from local data | v0.3.0 |
| `content_markdown` | Future: embedding input, RAG context, MCP responses | v0.4.0+ |
| `notion_page_id` | Reserved for future append/replace logic | Reserved |
| `notion_page_url` | "Open in Notion" link in preview | v0.3.0 |
| `created_at` | Sort order, relative date display, "last week" queries | v0.3.0 |
| `embedding` | 768-dim vector (3 KB BLOB), semantic search | v0.4.0 |
| `embedding_model` | Track model for re-embedding if model changes | v0.4.0 |

### TypeScript types

```typescript
export interface RecentNoteRow {
  id: string;
  title: string;
  branchName: string;
  createdAt: number;
}

export interface FullNote {
  id: string;
  title: string;
  branchName: string;
  content: StructuredNote;
  contentMarkdown: string;
  notionPageId: string | null;
  notionPageUrl: string | null;
  createdAt: number;
}

export interface NoteMetadata {
  branchName: string;
  notionPageId: string;
  notionPageUrl: string;
}
```

### Queries (6 total)

1. **INSERT** — on sync success
2. **LIST** — `SELECT id, title, branch_name, created_at FROM notes ORDER BY created_at DESC`
3. **GET ONE** — `SELECT * FROM notes WHERE id = ?`
4. **CLEAR ALL** — `DELETE FROM notes`
5. **EXPORT ALL** — `SELECT * FROM notes ORDER BY created_at DESC`
6. **SCHEMA VERSION** — `SELECT MAX(version) FROM schema_version`

### Migrations mechanism

```typescript
const MIGRATIONS: { [version: number]: (db: SqlDatabase) => void } = {
  1: (db) => { /* CREATE TABLE notes, schema_version, index */ },
  // 2: (db) => { /* v0.4.0 embedding backfill */ }
};
```

Runs at `init()`. Checks current version, applies any unapplied migrations in order, persists.

### Persistence strategy

| Action | Persist to disk? |
|---|---|
| Insert a note | Yes — immediately after INSERT |
| Clear all | Yes — immediately after DELETE |
| Read (list / get one) | No |
| App startup | No — disk → memory only |

### Storage math

- v0.3.0 note: ~3 KB (no embedding)
- v0.4.0+ note: ~6 KB (with embedding)
- 1 year heavy use (1500 notes): ~9 MB
- 5 years heavy use: ~45 MB
- Smaller than a single photo on your phone.

## User Flows

### Flow 1: First launch after upgrade from v0.2.0

1. Extension activates → `MemoryStore.init()` creates empty `devnote.db`
2. Credentials carry over from v0.2.0 → skip Setup
3. Drafts carry over → banner shown if one exists
4. Idle state shows Generate Doc button + empty Recent Notes section
5. Empty state message: "Your notes will appear here once you generate your first one."
6. No backfill from Notion — fresh start for local memory

### Flow 2: Generating a new note (happy path)

Steps 1-7 identical to v0.2.0. New steps after sync success:

8. `memoryStore.saveNote(currentNote, { branch, notionPageId, notionPageUrl })` writes row to SQLite
9. Success state shown (same as v0.2.0)
10. User clicks "Back to Generate Doc" → Idle → list refreshes → new note is the first row

### Flow 3: Clicking a past note

1. User clicks a row in Recent Notes list
2. Webview sends `{ type: 'clickRecentNote', id }` to extension
3. SidebarProvider calls `memoryStore.getNoteById(id)` → returns FullNote
4. Preview state renders with `isHistorical: true` flag
5. Preview shows same content layout as new-note preview, but different buttons:

| Element | New note preview | Historical preview |
|---|---|---|
| "Back" at top | Goes to Form | Goes to Idle |
| "Save Note" | Shown | Hidden |
| "Discard" | Shown | Hidden |
| "Open in Notion" | Not shown | Shown (subtle link at bottom) |

6. User clicks "Back" → returns to Idle → list intact

### Flow 4: Clearing all memory

1. User opens Settings (gear icon)
2. "Clear all memory" link at bottom of Settings (red text)
3. Click → confirmation popup:
   - Warning icon + "Clear all memory?" headline
   - Body: "This will permanently delete all notes DevNote has stored locally. Your Notion pages will NOT be touched."
   - Checkbox: "Save a backup JSON file first" (unchecked by default)
   - Buttons: [Cancel] (focused) + [Clear all] (red)
4. Cancel → popup closes, nothing happens
5. Clear all without export → DB wiped → toast: "Memory cleared. Your Notion pages are untouched."
6. Clear all with export → save dialog → JSON file saved → THEN DB wiped → toast with path
7. Save dialog cancelled → abort, nothing deleted (critical safety guarantee)

## Error Handling

### Guiding principle

> Notion is authoritative. Memory is best-effort. Never let a memory failure block a successful sync.

### Error matrix

| Error | Severity | User sees | Extension behavior |
|---|---|---|---|
| WASM won't load | Critical | Inline message in Recent section: "Memory unavailable" | v0.2.0 features work, memory disabled |
| DB corrupted | Critical | Toast + fresh start | Old file renamed to `.corrupt-{timestamp}`, new empty DB |
| Memory save fails after sync | Non-critical | Toast warning + normal success state | Note is in Notion, just not in local list |
| List won't load | Non-critical | Empty state + toast | Generate Doc still works |
| Note not found on click | Non-critical | Toast + list refresh | Stay in Idle |
| Disk write fails | Non-critical | Toast | In-memory state correct, next write retries |
| Clear fails | User-action | Toast | Nothing deleted, retry available |
| Export fails | User-action | Toast | Clear aborted, nothing lost |
| Save dialog cancelled | User-action | Silent | Nothing happens |

### Priority ladder

```
Notion sync succeeds?
├── YES → Memory save succeeds?
│         ├── YES → Success state (green)
│         └── NO  → Success state (green) + toast warning
└── NO  → Sync error state (red) + draft saved
```

Memory failure never overrides Notion success.

## Migration from v0.2.0

### What carries over automatically

- Gemini API key (SecretStorage) — untouched
- Notion token (SecretStorage) — untouched
- Notion database ID (workspace config) — untouched
- Draft data (globalState) — untouched

### What's new

- `globalStorageUri/devnote.db` — created empty on first launch
- Recent Notes section — shows empty state until first note is generated

### No backfill

Existing Notion pages from v0.1.x/v0.2.0 are NOT imported. Reasons:
- Parsing Notion blocks back to StructuredNote is lossy
- Clean mental model: "Recent Notes shows everything from v0.3.0 onward"
- Can be added as optional import command in a future release if users ask

## What v0.3.0 is NOT

- NOT semantic search (v0.4.0)
- NOT chat/RAG queries (v0.5.0)
- NOT MCP memory server (parked for post-v0.5.0)
- NOT editing past notes (read-only; edit in Notion)
- NOT deleting individual notes (only "Clear all"; individual delete if users ask later)
- NOT syncing back from Notion (one-way: DevNote → Notion)
- NOT Notion backfill (fresh start from v0.3.0 onward)
